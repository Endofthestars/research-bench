# 证据基座读取协议(auditor / strategist 共用)

> 两个服务的所有判断都必须建立在同一套可审计的外置证据上。本协议规定:读哪些文件、
> 数据源优先级、过期判定。两个 agent 定义各自引用本文件,不重复内联。

## 1. 读取清单(按序)

1. **config manifest**:项目 `.claude/research-bench.config.md` 顶部 frontmatter,
   确认 `modules`(无 config → 停止,让主流程提示运行 `rf:init`;Claude Code 中带前导 `/`)。
2. **§3 指标**:主/次/OOD 指标名与计算来源——一切定量表述只许用这里已有的指标,禁止自行定义。
3. **§2 架构地图文档**:当前架构现状(哪些模块已改、哪些 flag 已验证、权重兼容性结论)。
4. **§10 方向/路线注册表**(仅当 manifest 含 `directions`,否则按 §4 降级):
   - 活跃方向注册表——在研方向的单一事实来源,一行一个 slug;
   - 方向文件目录——各方向的「子任务表」(= 计划执行的实验,plan-of-record)、
     「结果表」(= 已回填结果)、「状态」(= 生命周期);
   - 教训库(已关闭方向的硬约束)——新假设的前提;
   - 结果汇总表(如 `ablation_table.csv`)——已完成实验的定量结果。
5. **§9 实验跟踪**(仅当 manifest 含 `tracking`):查看地址、run 的记录字段约定。
6. **§12 Zotero 文献库**(经 MCP 检索;仅当 manifest 含 `discovery`):MCP 端点与集合约定
   (`directions/<slug>` 每方向一集合)——文献支撑优先引库内已验证条目;
   引用「已验证」的定义与降级核验见 `novelty-protocol.md` §2。

## 2. 数据源优先级

- **结果汇总表为主数据源**;不假设跟踪系统实时可达。
- 需要实时/精细 run 数据(如区分 smoke 与全量的 epochs 参数)时,**告知主流程让用户去
  §9 查看地址核对后粘贴**,不自行猜测、不虚构 run。
- 用户当场粘贴的数据 > 汇总表 > 记忆/印象(后者禁止作为证据)。
- **文献维度**:用户粘贴的论文段落 > Zotero 库内已验证条目 > web 检索结果
  (检索到但未落库的一律视为未验证,只能作线索,见 `novelty-protocol.md` §2)。

## 3. 过期判定

命中任一即判「过期」,结论必须显式标注"基于可能过期的数据":
- 结果汇总表落后:跟踪系统 / 用户口述中存在已完成 run,但汇总表无对应行
  (修复:主流程直接执行轻 op `collect`,见 config §7.2)。
- 方向文件落后:子任务标 ☐ 但已有对应 run,或「状态」与注册表行不一致。
- auditor 报告(作为 strategist 输入时)落后:报告生成时间早于其后任何一次
  launch / collect,或用户说明"之后又执行了实验"。

## 4. 模块缺席时的降级

- 无 `directions`:plan-of-record 改用**用户提供的消融清单 / 论文草稿表格**;
  跳过方向漂移核查与教训库比对,说明一句"未启用 directions 模块,跳过 X"。
- 无 `tracking`:auditor 无从核查完整性,应停止提示先用 init 启用;
  strategist 的定量证据改为以用户粘贴为准。
- 无 `discovery`:文献引用无 Zotero 落库验证路径,降级为三层核验 + `[UNVERIFIED]` 标记
  (见 `novelty-protocol.md` §2);跳过查新关卡产出的比对,说明一句"未启用 discovery 模块,跳过 X"。
