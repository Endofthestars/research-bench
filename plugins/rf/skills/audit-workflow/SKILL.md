# 审计工作流（只读快照 / 清单核对）

> **宿主兼容（必读）**：开始前读取 `../../references/host-compatibility.md`；报告保护机制时必须区分
> Claude Code 的 hook 强制拦截与 Codex 的模型级预检。

两种模式，开场按用户意图选择；无法判断时先询问一次：
- **轻模式(快照)**：只读当前工作流结构，不做清单核对和一致性判断。
- **清单模式(核对)**：按**唯一标准**（配置 §11 指定的工作流检查清单，如 `docs/workflow-checklist.md`）
  逐条核对工作流脚手架,报告漂移与可优化点。
两种模式都**只读、只报告，不实施任何修改**；修改统一走 update-workflow 并等待用户确认。

> **所属模块**:maintenance;必需 `maintenance`(检查清单与脚手架范围都在 §11);
> 可选 其余模块(轻模式展示内容按 manifest 裁剪,未启用模块对应的节直接省略并注明"未启用")。
> **执行方式**:主流程执行(只读)。
> **开场三步**:① 读项目 `.claude/research-bench.config.md` 顶部 frontmatter manifest,确认 `modules`;
> ② 无 config → 停止,提示先运行 `rf:init` 初始化(Claude Code 中为 `/rf:init`);
> ③ `modules` 不含 `maintenance` → 停止,提示先用 init 启用该模块(它会实例化检查清单骨架)。
> 确认后读 §11(检查清单路径、脚手架范围);轻模式再按启用模块读 §5(执行环境)、§9(跟踪)。

## 轻模式:结构化快照(只读全貌)

### 读取范围
- `.claude/skills/*/SKILL.md` —— 每个 skill 的 name + 职责一句话
- `.claude/agents/*.md` —— 每个服务的职责、工具、约束摘要
- `.claude/hooks/` + `.claude/settings.json` —— 有哪些保护机制、各自拦什么
- `.claude/research-bench.config.md` —— 项目专属值(执行档案、句柄、路径、端点、指标)
- `scripts/` —— 部署 / 运行 / op 脚本的用途与子命令
- 容器构建文件 —— 镜像分层关系
- `experiments/` —— 训练封装 / smoke / config 的角色

### 输出结构
1. **Skills**:表格,列 名称 / 职责 / 是否仅显式触发(含 Codex `agents/openai.yaml` policy)
2. **服务(agents)**:六类服务各自管什么、边界在哪(architect / operator / auditor / strategist / reviewer / steward)
3. **Hooks / 保护机制**:Claude guard hook 拦什么、Codex 对应预检规则、settings 关键项
4. **镜像分层**(docker 运行时档案;venv 档案改列 §5.3 环境定义):base → dev / run 的依赖关系与各自用途(据配置 §5.2)
5. **实验流程**:[build-env → deploy-env(按档案适用)→] run-experiment(关键 op 经 operator)→ 跟踪 的链路
6. **关键路径与句柄**:执行档案(exec-profile)、执行句柄(remote 档案)、工作目录、权重路径、数据集挂载、跟踪写入端点与查看地址(据配置 §5/§8/§9)
> 轻模式要求：如实反映文件**当前内容**，不凭记忆推断；不评判一致性，不提出修改建议。

## 清单模式:照清单核对(一致性审计)

### 流程
1. **读标准**:打开配置 §11 的检查清单,以它的检查项(C1…Cn)为准。
2. **逐条核对**:对每一项,执行其"怎么查"列出的命令/动作,得出 通过 / 漂移 / 不适用。
   - **检查标准均以清单为准**；若发现缺项，只提出"建议给清单加项"，不得临时增加隐藏判据。
3. **额外建议**(基于读到的全貌,清单之外的优化):冗余/重复、职责重叠、保护机制缺口、过时占位等。
4. **输出报告**（见下），由用户决定是否进入 update 流程。

### 报告组织(按严重度)
1. 🔴 **会导致执行错误**:文件名/镜像名/路径/执行句柄漂移、占位符未替换、hook 漏拦训练入口、ops 映射缺失/脚本不存在、配置 § 值与实际代码不符等
2. 🟡 **矛盾或缺口**:自相矛盾的指令、引用失效文件、职责重叠、保护机制覆盖不全
3. 🟢 **可选优化**:冗余合并、措辞统一、清单建议新增项

每条包含：命中的清单项（C 编号）、证据（文件:行 或命令输出）、影响和建议修复方式（交由 update 执行）。

## 规则(强制,两模式通用)
- **只读**:不得写/改任何文件。发现漂移只报告,不直接修。
- 标准唯一(清单模式):一致性判据全部来自配置 §11 的检查清单,audit 不自带隐藏判据。
- 与 update 共用同一份清单 —— steward 改完的自检复用这里的核对逻辑。
- 守 steward 通用约束:不修改模型源码(配置 §2)、不修改远程运维。
