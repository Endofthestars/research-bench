# 设计文档：research-bench 自动化研究流水线

> 本文档汇总 2026-07 的需求讨论结果，作为后续开发的设计依据。
> 所有「已确认」条目均经用户逐条确认；「待细化」条目在对应迭代实施前需再确认。
> **现状标注更新至 research-bench 拆分后**（2026-07）：仓库已从单插件演进为 `rf` / `ef`
> 双插件结构，并新增可选的 `remote-control` 远程触发服务；历史章节保留原始设计脉络。

---

## 1. 需求本质

将深度学习实验流程标准化：用户提供一个研究想法后，agent 自动完成
调研 → 制定 plan/todolist → 执行（改码/执行实验/验证）→ 分析数据 → 给出指导建议。

**角色模型：agent 推进流程，人负责审核。** 人的介入点由档位决定（见 §3.1），
审核方式从「事前逐步确认」后移为「git commit 审计 + 自动验证保障」。

```
研究想法
   ↓
【调研】读源码 / 现有结果 / 文献，定位 idea 落点
   ↓
【规划】plan + todolist（改哪里、执行哪些实验、如何设计对照）
   ↓
【执行】按档位自动：改码（git 子分支）→ smoke 验证 → 启动正式训练
   ↓
【看护】探针 + 看门狗定时检查，训练期间 agent 不长期占用会话
   ↓
【分析】训练结束被唤醒/恢复执行 → 分析 agent 读数据 → 报告 + 下一步建议
   ↓
（loop 档位下建议自动成为下一轮输入，共 n 轮）
```

v0.4.0 时形成的四层模型（意图/服务/驱动/机制）仍是流水线的基础方法论；当前实现已拆分为
`rf` / `ef` 两个真实插件，并把可远程触发的会话入口放入 `remote-control` 可选服务。
仍需补齐**编排层**（接收研究想法、生成 todolist、连续推进、状态可恢复）
和**可靠性基础**（四类问题，见 §4；其中两项已在 0.2–0.4 基本解决，见表内标注）。

### 1.1 双重身份与阶段

| | 阶段一 | 阶段二 |
|---|---|---|
| 目标 | 在当前研究项目中完成流水线验证并稳定运行 | 完善 onboarding，面向中文 ML 社区开源发布 |
| 优先级 | 自用体验优先 | 新用户安装后可按文档完成初始化 |
| 约束 | 设计决策需要保留阶段二扩展空间，避免只适用于单一环境的硬编码 | |

- 语言：**继续中文**（skill / 文档 / 模板）。
- 能力范围：**暂不扩展**——论文产出支持、多人协作等留待后续版本。
  （实验监控的需求被探针-看门狗机制吸收，不单独做监控 skill。）
- 阶段验收标准：在对应迭代细化时确定。

## 2. 三层架构（延续原设计，本次补完）

| 层 | 载体 | 归属 | 现状 → 目标 |
|---|---|---|---|
| 方法论层 | skills / agents / hooks | plugin，随版本升级 | 四层模型已定型（0.3）→ 补编排层入口 skill |
| 项目事实层 | `.claude/research-bench.config.md`（**保持 markdown**） | 项目 | ~~手填模板~~ → **init 已实现**：预扫描（0.3.1）+ 两问定档（0.4）+ 模块装配 + 契约探测；`config` skill 可非交互装配。剩余工作是生成项目专属驱动脚本 |
| 确定性执行层 | 项目专属脚本（驱动/checklist/测试/探针/看门狗/回写） | 项目，向导生成 | ~~完全缺失~~ → **部分就位**：test-workflow.sh / run.sh / checklist 骨架由 init 实例化，执行序列由 ISA+ops 映射承载；**仍缺**：探针 / 看门狗 / 回写脚本 |

### 2.1 编排层构件 ↔ 既有基座对照

| v2 需要 | v0.4 已有的原语 | 还缺什么 |
|---|---|---|
| 执行序列（同步→装包→smoke→启动） | `operator` 服务 + 七 op ISA + ops 按档案预设 + 租约互斥 + 脱敏回执 | 断点恢复执行（状态文件驱动） |
| setup 向导（§4 问题 1） | `init`（预扫描/两问定档/装配/契约探测）+ `config init` 非交互 | 驱动脚本生成 |
| 调研/分析组件 | `architect`（架构）、`auditor`（完整性）、`strategist`（假设），auditor→strategist 单向流 | 编排层将它们组织成一轮流程 |
| 安全模型(§3.4) | `guard-train-channel`（受控通道,随 exec-profile）+ `guard-protected-write`（源码/保护机制写→ask） | gated/semi 档的人工确认用 ask 原语即可,无新机制 |
| 看护(§3.2) | 租约文件已标识「在执行实验」,可作探针的锚点之一 | 探针/看门狗/唤醒全部 |
| 状态可恢复(§3.5) | config manifest 是静态事实来源 | `pipelines/<slug>/run-<n>/` 动态状态文件全部 |
| **注意** | 探针/看门狗设计需按 **exec-profile 四档案**适配（local 直读文件,remote 经句柄取回) | |

**核心原则（已确认）：**
- 判断交给模型，裁决与执行交给脚本。
- 生成优于内置：plugin 不内置项目运行时；向导采用「**模板引导 + 模型生成**」方式。
  仓库预留**示例级**模板，用于展示推荐产物结构，不承诺直接可运行。
  无匹配模板时，模型根据探测和问答结果生成。环境通用性依赖生成灵活性，不依赖硬编码适配器。
- plugin 对 train_wrapper / smoke_test 等实验脚手架**只带接口约定文档**，实现属于项目。
- **保护机制不可自动削弱**约束保持不变。

## 3. 流水线设计（已确认决策）

### 3.1 档位体系（四档）

| 档位 | 计划确认 | 改码写入文件 | 正式训练启动 | 下一轮迭代 |
|---|---|---|---|---|
| `gated` | 人工确认 | 人工确认 | 人工确认 | 人工确认 |
| `semi`（默认） | 人工确认 | 自动 | 人工确认 | 人工确认 |
| `auto` | 自动 | 自动 | 自动 | 人工确认 |
| `loop(n)` | 自动 | 自动 | 自动 | 自动执行 n 轮，n 为参数，受预算上限约束 |

### 3.2 时间跨度：探针 — 看门狗 — 唤醒

正式训练可能持续数小时到数天，agent 不应长期保持会话等待。一轮流水线的自然断点 =
「训练已启动 + 看门狗已交接」。

- **探针**：由训练封装注入，持续写**独立的本地 JSON 状态文件**
  （心跳、epoch 进度、最新指标、异常栈）到约定位置。
  不依赖 MLflow 可用性/网络（崩溃、卡死时 MLflow 未必有信号）；深度指标分析仍用 MLflow。
- **看门狗**：定时任务（system cron 等）调用**确定性脚本**读探针，按固定逻辑判定：
  正常训练 → 不打扰；结束 / 崩溃 / 指标异常 → 触发唤醒动作。
- **唤醒**（迭代②）：先只通知用户（渠道待定），用户确认继续后，流水线从状态文件恢复。
  接口解耦：看门狗只执行 config 里的一条 `WAKE_CMD`；
  升级路：`claude -p` headless 自动拉起 → Agent SDK 常驻 daemon（socket 监听）。
  Claude Code 运行中会话无公开 socket 注入接口；claude.ai 云端触发够不到内网，排除。
- **会话策略**（倾向，未强制确认）：唤醒/恢复执行时创建**全新会话冷启动**，
  从 状态文件 + 探针数据 + git 子分支 恢复全部上下文。
  这会要求状态文件成为完整事实来源，有利于后续恢复和审计。

### 3.3 git 模型：方向分支 + 流水线子分支

- 每个研究方向一个 slug + **方向分支**；每轮流水线在方向分支上开**子分支**。
- 所有自动改动一改一 commit 进子分支；实验跟踪记录锁子分支 hash。
- 方向成立 → 人审后并回方向分支；失败 → 整个子分支丢弃。

### 3.4 安全模型

改源码**不设事前人工关卡**（gated 档除外），安全性来自：
1. git 可回滚（§3.3，一改一 commit）；
2. 自动验证保障（强制）：预训练权重加载验证、smoke test、
   flag 全关时数值复现 baseline（等价性检查）；
3. guard hook 保护机制不变（0.4 起为 `guard-train-channel`：绕开受控启动通道的训练照拦，
   通道形态随 exec-profile；另有 `guard-protected-write` 把写源码/写保护机制降级为用户当场确认）。

### 3.5 状态载体

- 位置：`experiments/pipelines/<slug>/run-<n>/`，提交进外层工作区仓。
- 内容：`plan.md`（人可读计划）+ `todo.yaml`（机器可读清单：步骤/状态/产物指针）。
- 断线、compact、换机器、审计全部依赖它；每完成一步立即写入文件更新。

### 3.6 用户入口

**多个独立 skill**（延续现有「职责单一」风格），暂定动作集：
开新一轮（携 idea 与档位参数）/ 恢复执行 / 查询状态 / 终止。命名待细化。

## 4. 可靠性基础（四类问题，随迭代按需补齐）

| 问题 | 已确认解法 | v0.4 状态 |
|---|---|---|
| onboarding 成本高 | setup 向导：自动探测 + 交互问答 + 生成全套（config、保护机制 env、checklist、测试脚本、驱动脚本）；模板引导 + 模型生成 | **大部分已实现**：init 预扫描+两问定档+装配+契约探测；config 可代写保护机制 env；缺驱动脚本生成 |
| 实验闭环不稳定 | 同步→装包→smoke→启动 固化进**驱动脚本**（自带断点恢复执行），skill 负责调用脚本、解读结果和处理分支 | **部分实现**：序列由 operator+ISA+ops 映射承载，skill 已保持轻量编排入口；缺断点恢复执行 |
| 方向管理繁琐 | 方向文件/注册表/结果回填三处**回写自动化**（机制待细化：脚本 vs skill） | 未动(迭代③) |
| 检查标准缺失 | checklist 与 test-workflow.sh 由向导生成**项目专属版**，audit/test skill 安装后即可使用统一判据 | **已实现**：骨架 C1–C10 + guards/ops 层可直接使用，init 实例化 |

## 5. 实施路线：编排层 MVP 先行，三步迭代

**迭代①：流水线骨干（无自动看护）**
- 编排入口 skill + 状态文件（§3.5）+ `semi` 档
- 链路：idea → 调研 → plan/todolist → 改码（子分支+commit）→ 自动验证 → smoke →（人工确认）启动 → 交接
- 训练监控先由用户触发恢复执行；复用现有 skill 作为流程组件
- 同步补齐：驱动脚本最简版（可恢复执行的 同步→装包→smoke→启动）
  ——v0.4 注：序列本身已由 operator+ISA 承载,本项收窄为「给这条序列增加断点恢复执行能力(读状态文件跳过已完成 op)」

**迭代②：探针 + 看门狗**
- 训练封装注入探针（独立 JSON）；看门狗脚本 + 定时任务；判定后**通知人**
- 补齐 `gated` / `auto` 档

**迭代③：闭环与自动化收尾**
- `loop(n)` 档（分析建议自动进下一轮，预算上限约束）
- 回写自动化（方向文件/注册表/结果回填）
- 可选：`WAKE_CMD` 升级为 headless 自动拉起

基础能力（setup 向导完整版、checklist/测试脚本生成、接口约定文档）根据主流程暴露的问题按需插入。
