# 细化研究方向（种子想法 → 多轮收敛 → 一个 slug）

**定位：** 收敛型、交互式方向细化入口。它与 `propose-hypothesis` 互补：`propose-hypothesis` 负责发散生成候选，本 skill 负责从**一个**种子出发，经主流程多轮对话逐维确定，最终产出**一个**「方向落地块」。
本 skill 只读、只产出文本，**不修改任何代码或文档**；写入文件交由 `update-workflow`。

> **所属模块**:directions;必需 `core`、`directions`(本 skill 的产出就是方向落地块,离开注册表无意义);
> 可选 无。**执行方式**:主流程多轮执行(不委托服务,见下)。
> **开场三步**:① 读项目 `.claude/research-bench.config.md` 顶部 frontmatter manifest,确认 `modules`;
> ② 无 config → 停止,提示先运行 `/rf:init` 初始化;
> ③ `modules` 不含 `directions` → 停止,提示先用 init 启用该模块。
> 确认后读 §2–3(源码/权重/指标)、§10(方向注册表、教训库、方向文件目录、slug 约定、方向轴取值)。下文用 `<占位>` 指代这些值。

## 执行方式(关键:不走子代理)
- **在主流程对话里直接执行，不委托任何服务**。该流程依赖多轮确认：用户提供引导，主流程对照证据提出补全或质疑，用户确认后继续收敛。
- 每一轮聚焦**一个待定维度**,用 `AskUserQuestion` 给选项 + 推荐,让用户确认;确认后进下一维。
- 这正是与 propose-hypothesis 的分界:propose 发散(strategist 服务,广度),refine 收敛(主流程,深度)。

## 使用时机
- 已有一个粗想法 / 一句"我想试试 X",想细化成正式方向
- 从 `propose-hypothesis` 的 3–5 张卡片里选择一张,要把它完善到可落地
- 几个想法纠缠不清,想通过对话收敛到单一 slug

## 输入收集(开场一次性读,锚定讨论)
1. **用户的引导/种子** —— 粗想法、选中的假设卡片、或一篇论文的关键段落(调用时带入)
2. **配置 §10 的方向注册表 + 教训库(已关闭方向的硬约束)** ——
   **避免重复与重复证伪**：新 slug 不得与注册表已有方向重复；新想法若触及教训库硬约束，
   需要当场提示用户，要么避开，要么说明为何能够突破。
3. **配置 §10 的方向文件目录** —— 进行中的方向的核心诊断与子任务,避免职责重叠
4. **配置 §2 的架构地图文档** —— 锚定"现状瓶颈"引哪个模块/行号;权重兼容性结论
5. **配置 §10 的结果汇总表**(按需)—— 用当前定量结果支撑"现状瓶颈"判断
6. **方向 dossier 的 novelty.md / review.md 产出**(若存在;启用 `discovery` 且该种子已过关卡)——
   对照表的差异点约束方向定位,薄弱论点与实验设计建议直接喂给「可证伪预测」维度;
   未启用或未过关时跳过本项,不阻塞收敛

## 收敛流程(逐维确定,每维一轮对话)
按下序逐维收窄;每维给 2–4 个带推荐的选项,用户确认后再进下一维。已明确的维度可跳过。
1. **现状瓶颈** —— 锚定要打的是哪个指标维度、数据流根因在哪(引架构地图文档)。先对齐"打什么"。
2. **教训库冲突检查** —— 比对教训库：是否已被证伪或受硬约束？若存在冲突，需先处理（避开或论证突破），否则不继续。
3. **改动假设** —— 干预作用在哪个模块(decoder / 输出头 / loss / 流场 / 特征分支),为什么能改善瓶颈。
4. **slug 命名** —— kebab,且**不与注册表已有 slug 冲突**;它将贯穿 config 名 / 跟踪 tag / commit 前缀 / 文件名。
5. **flag 设计** —— `<slug>_<param>`(>1 knob 时),**默认 off**,`flags:{}` 必须数值复现基线(纯加性开关)。
6. **baseline_group** —— 配置 §10 的取值之一(新方向选默认那个)。
7. **access_level** —— 配置 §10 的取值之一;碰输出头/backbone(高危)须标注"先过 architect 评权重兼容"。
8. **可证伪预测** —— 若假设成立,配置 §3 哪个指标朝哪个方向变、量级多少;以及失败模式。

## 收敛判据(满足才算"可落地")
- 核心诊断三件套(现状瓶颈 / 改动假设 / 可证伪预测)全部填满且具体;
- slug / flag / baseline_group / access_level 全部确定,slug 与 flag 命名空间均不与现有名称冲突;
- 不违反教训库任一硬约束(或已显式论证为何能突破)。
> 任一未满足 → 继续对话收敛,**不要**提前产出落地块。

## 产出:方向落地块(收敛完成后产 1 份)
与 `propose-hypothesis` 的「方向落地块」**同格式**——**字段与格式以
`${CLAUDE_PLUGIN_ROOT}/templates/project/directions/_TEMPLATE/direction.md` 为唯一模板**(项目内实例见
配置 §10 的方向文件模板):按模板字段逐项填出(含 frontmatter `status`(按实际关卡进度取值,
状态机见模板)、注册表行、首批子任务表),可由 `update-workflow` 直接粘成新方向 dossier + 登记注册表。

## 与上下游对接
```
[propose-hypothesis 选中一张卡片] 或 [用户直接带种子]
   → [启用 discovery 时先过三关:check-novelty(关卡 1)→ pilot(关卡 2,当前人工判,
      跳过须留 gates.jsonl 记录)→ reviewer 审查(关卡 3,两段喂料:先核心主张取盲评初判、
      再完整 dossier 取全档终判,协议见 reviewer 定义)]
   → refine-direction(本 skill,主流程多轮收敛,吸收 novelty.md / review.md 产出)→ 方向落地块
   → update-workflow(落方向 dossier + 登记注册表)
   → [modify-architecture(若需改源码)]
   → run-experiment(新 config 必填 direction:<slug>)
```

## 约束
- **只读、只产文本**:不改任何源码、配置或 docs 文档;写入文件统一走 `update-workflow`。
- **不修改模型源码**(architect / modify-architecture 的职责范围),不修改远程运维。
- **不替用户确认**:每维给选项 + 推荐,最终方向由用户定;不擅自跳维或越过收敛判据。
- **不创造指标**:可证伪预测只用配置 §3 已有指标。
- **遵守 slug 约定**(配置 §10):slug=kebab、新 flag=`<slug>_<param>`、默认 off、`flags:{}` 复现基线。
