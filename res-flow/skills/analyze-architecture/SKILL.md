# 架构分析（委托 architect）

将分析任务委托给 **architect 服务**（只读、隔离上下文），按固定流程产出可复用结论。

> **所属模块**:core;必需 `core`;可选 无。**执行方式**:委托 `architect` 服务。
> **开场三步**:① 读项目 `.claude/research-bench.config.md` 顶部 frontmatter manifest,确认 `modules`;
> ② 无 config → 停止,提示先运行 `/rf:init` 初始化;
> ③ `core` 必在(init 强制),确认后读 §2–3(`SOURCE_DIR`、关键源文件、预训练权重名、指标、架构地图文档)取值。

## 流程
1. **委托 architect 服务**调研相关源码(配置 §2 的 `SOURCE_DIR` 下)
2. 让它产出:
   - 相关文件与函数定位
   - 当前数据流(输入 → backbone → decoder →(流场)→ loss)
   - 若用户有具体改动意图:改动点、对预训练权重加载的影响、最小改动方案
   - 「可安全改造」vs「高危(破坏预训练)」的标注
3. 结论沉淀到配置 §2 的架构地图文档(architect 负责写;其记忆规则保证事实在文档、不在记忆)
4. 主对话只接收摘要，不返回整份源码内容

## 关键判断原则
- 若框架只支持微调、不支持从头训练（见配置 §1 约束），任何破坏预训练权重加载的改动均为高风险
- 优先把创新放在配置 §2 的「创新优先区」(decoder / 输出头 / loss / 流场),backbone 尽量保留
- 评估改动时，首要问题是：“官方预训练权重是否仍可加载？”

## 输出给用户
输出一份简明结论，说明建议修改位置、修改方式、风险和对预训练权重加载的影响。
